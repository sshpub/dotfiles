Three Approaches to the Dotfiles Overhaul

Each approach addresses the same goals — they differ in how much changes at once and where complexity lives

A

Modular Monorepo — Recommended

Core: Thin shell foundation (platform detection, PATH, prompt, exports, sync-check)

Modules: Self-describing directories with module.json — aliases, functions, install recipes. CLI discovers and loads them.

CLI: Expand Go CLI as the primary tool. Pre-built binaries per platform/arch behind a thin shell wrapper. Handles bootstrap, module management, sync, profile wizard.

Profiles: Per-machine ~/.dotfiles.json selects role + modules. Generated interactively on first run.

Claude: Hierarchical CLAUDE.md per directory + .claudeignore for large files.

Pros

  • Clean separation — modules are self-contained and shareable
  • Incremental migration — move things into modules one at a time
  • Module marketplace-ready from day one
  • Claude gets scoped context naturally

Cons

  • More files/dirs than today (but each is smaller and focused)
  • Shell startup adds a module discovery loop
  • Go CLI is the critical path — needs to be solid
B

Layered Config (Cascade Model)

Structure: base/ → platform/{macos,linux,wsl}/ → role/{work,personal}/ → machine/{hostname}/

Loading: Each layer overrides/extends the previous. Shell sources them in order.

CLI: Same Go CLI approach, but the CLI's job is simpler — just source the right layers in order.

Modules: No formal module system. Features are organized by layer instead.

Pros

  • Very predictable — "which layer set this?" has a clear answer
  • Natural for platform/role differences
  • Simpler mental model for override behavior

Cons

  • Features get split across layers (k8s aliases in base, platform, AND role)
  • No marketplace path — features aren't self-contained units
  • Adding a new tool means touching multiple layers
  • Harder for Claude — context scattered across layers
C

Hybrid — Modules + Light Layering

Same as A but modules themselves can have platform/role overrides internally:

modules/kubernetes/aliases.sh (base)
modules/kubernetes/aliases.macos.sh (platform overlay)
modules/kubernetes/aliases.work.sh (role overlay)

Benefit: Platform/role differences are colocated with the module, not in a separate layer tree.

Pros

  • All of A's benefits
  • Platform/role specifics stay with the module
  • Very clean for marketplace — module ships with all its variants

Cons

  • More convention to learn (filename-based overlays)
  • Module loader is more complex
  • Overkill if most modules don't vary by platform
