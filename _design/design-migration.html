Design: Migration Strategy

Section 6 of 6 — How to get from current state to the new architecture

Guiding Principle

Your shell must work at every step. No big-bang migration. Each phase produces a working dotfiles setup that you can deploy to your 6 machines. If something goes wrong, you can stop at any phase and still have a functional system.

Migration Phases

1

Scaffold & Core

Create the new directory structure. Build the core loader with minimal mode and section guard system. The new .bash_profile sources core/ files. No modules yet — core just provides what the current flat files do.

  • Create core/, modules/, overrides/, profiles/, platform/, setup/, cli/, bin/
  • Extract platform detection into core/platform.sh
  • Build core/loader.sh with dotfiles_section() and minimal mode
  • Move essential PATH/exports/aliases/functions into core/
  • Move platform-specific code into platform/
  • New .bash_profile that sources the new structure
  • Test: shell works identically to before
2

Extract Modules

Break the current monolithic files into self-contained modules. Each module gets a module.json, section guards, and its own shell files.

  • Split .aliases (426 lines) into ~10 modules (git, docker, k8s, python, node, etc.)
  • Split .functions (1,385 lines) into corresponding modules + core essentials
  • Create module.json for each with sections, install recipes, platform support
  • Add dotfiles_section() guards to all shell blocks
  • Move config files (.gitconfig, .vimrc, .tmux.conf) into their modules
  • Core loader discovers and loads modules from modules/
  • Test: all aliases/functions still work, now loaded via module system
3

Profile System

Implement the profile-driven loading. Machines can now choose which modules to enable.

  • Define ~/.dotfiles.json schema
  • Loader reads profile, builds enabled/disabled lists
  • Implement shell cache (CLI generates sourceable file from JSON)
  • Implement minimal mode with env triggers
  • Create example profiles for your 6 machines
  • Test: different profiles produce different shell environments
4

Go CLI Overhaul

Rebuild the Go CLI from the existing dotfiles-cli. New command tree, module management, profile wizard, override system.

  • Restructure into cli/ with new command tree
  • Implement: setup, bootstrap, update, module *, profile *, doctor
  • Implement: override clone, section extraction
  • Implement: config conflict resolution (backup/merge/skip/adopt)
  • Implement: minimal mode CLI commands
  • Build shell wrapper (bin/dotfiles)
  • Cross-compile for all 4 targets
  • Optional: bubbletea TUI for module browse
  • Test: full setup workflow on a fresh machine
5

Registry & Sync

Add registry support for marketplace modules, sync checking, and multi-machine awareness.

  • Implement registry add/remove/list/sync CLI commands
  • Module search and install from registries
  • Private registry support (git SSH auth)
  • Sync check: background git fetch, cached state, shell notification
  • Test: install a module from a test registry
6

Claude Context & Cleanup

Add scoped CLAUDE.md files, .claudeignore, clean up old flat files, update documentation.

  • Write hierarchical CLAUDE.md files for each directory
  • Create .claudeignore
  • Remove old flat files (.aliases, .functions, .platform, etc.)
  • Update README.md for new architecture
  • Write MODULE-SPEC.md for marketplace authors
  • Update bootstrap.sh symlink list
  • Test: full deploy on all 6 machines

Rollback Safety

Git Branch

All work on a feature branch. Main stays working. If anything breaks: git checkout main

Phase Gates

Each phase is a merge to main. Deploy one phase at a time to machines. Verify before next.

Backward Compat

Phases 1-3 can coexist with old files. Old .bash_profile still works until phase 6 cleanup.

Deployment Order

Roll out to machines incrementally
Phase 1-3: Test on 1 WSL machine first - Fastest to rebuild if something breaks - You're already on this machine (presumably) - Validate core + modules + profile system Phase 4: Deploy CLI to all machines - git pull gets them the new binary - Run 'dotfiles setup' on each to generate profile Phase 5-6: Roll out to remaining 5 machines - One at a time, verify each - Macbooks last (most different platform, most to test) Estimated module breakdown from current files: .aliases (426 lines) splits into ~12 modules: git, docker, kubernetes, cloud, python, node, navigation, modern-tools, safety, network, system, terraform .functions (1,385 lines) splits into: ~300 lines core/functions.sh (essentials) ~1,085 lines across modules (cloud, security, containers, etc.)
