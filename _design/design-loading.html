Loading Chain

Shell Startup Sequence
~/.bash_profile (symlink → dotfiles/.bash_profile) │ ├─ 1. CORE BOOTSTRAP │ ├─ core/platform.sh # Detect OS, arch, distro, pkg manager │ ├─ core/loader.sh # Define dotfiles_section(), load_module() │ ├─ core/path.sh # Base PATH (~/.local/bin, ~/bin) │ ├─ core/exports.sh # EDITOR, LANG, HISTSIZE, etc. │ ├─ core/aliases.sh # .., ll, la, mkd, reload │ ├─ core/functions.sh # extract, server, mkd, backup │ └─ core/completions.sh # Base bash completion setup │ ├─ 2. PLATFORM LAYER │ └─ platform/{detected}.sh # macos.sh, linux.sh, or wsl.sh │ # Clipboard (pbcopy/pbpaste), open command, │ # Homebrew setup, WSL interop, etc. │ ├─ 3. LOAD PROFILE │ └─ Read ~/.dotfiles.json # Which modules are enabled + disabled sections │ # Build DOTFILES_DISABLED_SECTIONS array │ # Build DOTFILES_ENABLED_MODULES list │ ├─ 4. MODULE LOADING (for each enabled module) │ │ │ │ modules/{name}/ │ │ ├─ a. Read module.json # Check platform compatibility │ │ ├─ b. Source aliases.sh # With dotfiles_section() guards │ │ ├─ c. Source functions.sh # With dotfiles_section() guards │ │ ├─ d. Source completions.sh # If present │ │ └─ e. Add to PATH # If module defines paths │ │ │ └─ Repeat for each enabled module │ ├─ 5. OVERRIDES │ ├─ overrides/*.sh # Repo-level (tracked, all machines) │ └─ ~/.dotfiles/local/*.sh # Local-level (untracked, this machine) │ ├─ 6. PRIVATE CONFIG │ ├─ ~/.extra # Secrets, API keys, work credentials │ └─ ~/.local # Additional local config │ ├─ 7. PROMPT │ └─ core/prompt.sh # Oh My Posh / Starship / fallback prompt │ └─ 8. SYNC CHECK └─ core/sync-check.sh # Read cached state, show notification if behind # "dotfiles: 3 commits behind (run 'dotfiles update')"

The Section Guard System

How dotfiles_section() works
# Defined in core/loader.sh # DOTFILES_DISABLED_SECTIONS is populated from ~/.dotfiles.json dotfiles_section() { local section="$1" # Check if section is in the disabled list [[ ! " ${DOTFILES_DISABLED_SECTIONS[*]} " =~ " ${section} " ]] } # Usage in a module: dotfiles_section "git.shortcuts" && { alias g='git' alias gs='git status' alias ga='git add' } dotfiles_section "git.log" && { alias glog='git log --oneline --graph' alias glast='git log -1 HEAD' } # If "git.shortcuts" is in DOTFILES_DISABLED_SECTIONS, # the entire block is skipped — never parsed, never executed.

Profile-Driven Loading

~/.dotfiles.json drives what loads
{ "role": ["personal", "work"], "modules": { "git": true, // shorthand: shell + install "kubernetes": { "shell": true, "install": true, "disable": ["kubernetes.helm"] // skip helm section }, "cloud": { "shell": true, "install": false }, "vim": true, "containers": true, "python": true, "node": true } } // Loader reads this at step 3, builds: // DOTFILES_ENABLED_MODULES=(git kubernetes cloud vim containers python node) // DOTFILES_DISABLED_SECTIONS=(kubernetes.helm)

Performance Budget

  • Target: Shell startup under 200ms (current lazy-loading approach preserved)
  • Profile parsing: CLI pre-generates a shell-sourceable cache from JSON (~1ms to source vs parsing JSON in bash)
  • Module discovery: Cached — CLI builds a module manifest on dotfiles update, not on every shell open
  • Section guards: Array lookup — negligible overhead per check
  • Sync check: Reads a cache file only — no network on shell startup
