# Profile System Design — Issue #19

**Date:** 2026-04-05
**Author:** Anton Swartz
**Status:** Approved

## Overview

The profile system reads a JSON profile (`~/.dotfiles.json` or other locations) and drives what gets loaded. A throwaway shell script generates a cache file that the loader sources at startup. The CLI (Phase 4) replaces the generator.

## Decisions

- **Cache generator:** Standalone `setup/generate-cache.sh` using jq. Throwaway — CLI replaces it.
- **No-profile default:** All bundled modules enabled (discover `modules/*/module.json`).
- **Profile search chain:** Env var override + conventional paths, first match wins.
- **Data directory search chain:** Env var override + conventional paths for `~/.dotfiles/` equivalent.
- **Modes system:** Generic `modes` map replaces hardcoded `minimal_mode`. First triggered mode wins. `minimal` has built-in defaults when no `modes` key is present.

---

## 1. Profile JSON Schema

All fields are optional. Missing profile = all modules enabled. Missing `modules` key = all modules enabled. Present but empty `modules: {}` = no modules.

```json
{
  "_comment": "Machine profile — all fields optional",
  "role": ["personal", "work"],
  "platform": {
    "_comment": "Auto-detected if omitted, override if needed",
    "os": "linux",
    "variant": "wsl",
    "distro": "ubuntu"
  },
  "modules": {
    "git": true,
    "kubernetes": {
      "shell": true,
      "install": true,
      "disable": ["kubernetes.helm"]
    },
    "cloud": { "shell": true, "install": false }
  },
  "git": {
    "name": "Anton Swartz",
    "email": "anton@work.com"
  },
  "modes": {
    "minimal": {
      "env_triggers": ["CLAUDE_CODE", "CODEX", "GEMINI_CLI", "CI"],
      "include_modules": ["git"],
      "never_load": ["prompt", "completions", "sync-check"]
    },
    "server": {
      "env_triggers": ["SSH_SESSION"],
      "include_modules": ["git", "safety", "system"],
      "never_load": ["prompt", "fzf"]
    }
  },
  "registries": [
    { "name": "default", "url": "https://github.com/sshpub/dotfiles-modules" }
  ]
}
```

### Module entry rules

- `true` shorthand expands to `{ "shell": true, "install": true }`
- `disable` array uses `module.section` format matching the module schema's sections
- `registries` is defined in schema but not processed until Phase 5
- `_comment` fields allowed anywhere

### Mode rules

- Each mode has: `env_triggers` (array of env var names), `include_modules` (modules to load in that mode), `never_load` (core components / modules to skip)
- `DOTFILES_MODE=<name>` env var forces a mode directly, bypassing trigger checks
- Modes are checked in declaration order — first triggered mode wins
- If no `modes` key in profile, loader uses hardcoded minimal defaults

### Formal schema

A JSON Schema file at `profiles/schema.json` validates the profile structure. It covers all fields above with proper types, patterns, and constraints.

## 2. Profile Search Chain

Both the generator and loader use this search order to find the active profile. First match wins.

1. `$DOTFILES_PROFILE` (env var override)
2. `~/.dotfiles.json`
3. `~/.config/dotfiles/profile.json`
4. `~/.config/dotfiles.json`
5. `~/.local/dotfiles.json`
6. `$DOTFILES_DIR/dotfiles.json`
7. `$DOTFILES_DIR/profiles/default.json`

If none found, no-profile default applies (all bundled modules enabled).

The resolved path is recorded in the cache as `DOTFILES_PROFILE_SOURCE`.

---

## 3. Data Directory Resolution

The data directory holds cache and local overrides. Currently hardcoded as `~/.dotfiles/`. Becomes configurable with the same pattern as profiles.

Search order (first existing directory wins):

1. `$DOTFILES_DATA_DIR` (env var override)
2. `~/.dotfiles/`
3. `~/.config/dotfiles/`
4. `~/.local/share/dotfiles/`

If none exist, defaults to `~/.dotfiles/` (created on first cache write).

Derived paths:
- `DOTFILES_CACHE_DIR="${DOTFILES_DATA_DIR}/cache"`
- Local overrides: `${DOTFILES_DATA_DIR}/local/*.sh`

The resolution function `_dotfiles_resolve_data_dir` lives in `core/platform.sh` since it's the first file loaded and already writes the platform cache.

## 4. Cache Format

Generated file: `${DOTFILES_CACHE_DIR}/profile.sh`

```bash
# Generated by dotfiles — do not edit
# Source: /home/necro/.dotfiles.json
# Generated: 2026-04-05T10:30:00

DOTFILES_PROFILE_SOURCE="/home/necro/.dotfiles.json"
DOTFILES_ENABLED_MODULES=(git kubernetes cloud python node modern-tools)
DOTFILES_DISABLED_SECTIONS=(kubernetes.helm)

# Modes — first triggered wins, checked in order
DOTFILES_MODE_NAMES=(minimal server)
DOTFILES_MODE_minimal_TRIGGERS=(CLAUDE_CODE CODEX GEMINI_CLI CI)
DOTFILES_MODE_minimal_MODULES=(git)
DOTFILES_MODE_minimal_NEVER_LOAD=(prompt completions sync-check)
DOTFILES_MODE_server_TRIGGERS=(SSH_SESSION)
DOTFILES_MODE_server_MODULES=(git safety system)
DOTFILES_MODE_server_NEVER_LOAD=(prompt fzf)
```

Variable naming convention: `DOTFILES_MODE_{name}_{TRIGGERS|MODULES|NEVER_LOAD}`.

`DOTFILES_MODE_NAMES` preserves declaration order for first-match evaluation.

When no profile exists, no cache is written — the loader handles the all-modules default inline.

## 5. Loader Changes (`core/loader.sh`)

### Default profile — all-modules discovery

Replace the current empty `DOTFILES_ENABLED_MODULES=()` fallback:

```bash
_dotfiles_default_profile() {
  DOTFILES_ENABLED_MODULES=()
  local d
  for d in "${DOTFILES_MODULES_DIR}"/*/; do
    [[ -f "${d}module.json" ]] && DOTFILES_ENABLED_MODULES+=("$(basename "$d")")
  done
  DOTFILES_DISABLED_SECTIONS=()
  DOTFILES_MODE_NAMES=(minimal)
  DOTFILES_MODE_minimal_TRIGGERS=(CLAUDE_CODE CODEX GEMINI_CLI OPENCODE GROK_CLI CI GITHUB_ACTIONS GITLAB_CI)
  DOTFILES_MODE_minimal_MODULES=()
  DOTFILES_MODE_minimal_NEVER_LOAD=()
}
```

### Mode resolution

New function replaces the minimal-only check:

```bash
dotfiles_resolve_mode() {
  # Env var override
  if [[ -n "$DOTFILES_MODE" ]]; then
    DOTFILES_ACTIVE_MODE="$DOTFILES_MODE"
    return 0
  fi
  # Check each mode's triggers in order
  local mode var
  for mode in "${DOTFILES_MODE_NAMES[@]}"; do
    local -n triggers="DOTFILES_MODE_${mode}_TRIGGERS"
    for var in "${triggers[@]}"; do
      if [[ -n "${!var}" ]]; then
        DOTFILES_ACTIVE_MODE="$mode"
        return 0
      fi
    done
  done
  DOTFILES_ACTIVE_MODE=""
  return 1
}
```

`dotfiles_is_minimal()` becomes a thin backward-compat wrapper:

```bash
dotfiles_is_minimal() {
  [[ "$DOTFILES_ACTIVE_MODE" == "minimal" ]]
}
```

### Mode-aware module loading

New helpers to get the active mode's config:

```bash
dotfiles_load_mode_extras() {
  local -n modules="DOTFILES_MODE_${DOTFILES_ACTIVE_MODE}_MODULES"
  local mod
  for mod in "${modules[@]}"; do
    load_module "$mod"
  done
}
```

## 6. `.bash_profile` Changes

Step 2 changes from `dotfiles_is_minimal` to `dotfiles_resolve_mode`:

```bash
# ─── 2. Mode Check ────────────────────────────────────────────
if dotfiles_resolve_mode; then
    dotfiles_load_mode_extras
    return 0 2>/dev/null || exit 0
fi
```

All active modes do an early return — only `include_modules` are loaded. `never_load` is defined in the schema and cache for forward compatibility but is **not enforced in this issue**. Issue #20 (minimal mode) will implement selective loading where `never_load` skips specific core components without an early return.

Step 7 local overrides use resolved data dir:

```bash
# Local-level (untracked, per-machine)
for _f in "${DOTFILES_DATA_DIR}"/local/*.sh; do
    _dotfiles_source "$_f"
done
```

## 7. Generator Script (`setup/generate-cache.sh`)

Throwaway jq-based script. Bridges the gap until the CLI (Phase 4).

**Requires:** jq (exits with clear error if missing).

**Behavior:**
- Finds profile via search chain (argument > env var > default paths)
- Expands `true` shorthand to `{ "shell": true, "install": true }`
- Builds `DOTFILES_ENABLED_MODULES` from modules where `shell` is true
- Collects `disable` arrays into `DOTFILES_DISABLED_SECTIONS`
- Iterates `modes` map, generates per-mode variables in declaration order
- Writes `${DOTFILES_CACHE_DIR}/profile.sh`
- If no profile found, prints message and exits (no cache needed)
- Idempotent — safe to run repeatedly
- Prints: profile path found, cache path written

**Usage:**
```bash
./setup/generate-cache.sh                              # auto-detect profile
./setup/generate-cache.sh ~/work-profile.json          # explicit path
DOTFILES_PROFILE=~/work.json ./setup/generate-cache.sh # env var
```

No schema validation — that's a CLI concern.

## 10. Out of Scope

- Schema validation at load time (CLI, Phase 4)
- Registry processing (Phase 5)
- Git identity from profile (in schema, no consumer yet — follow-up or Phase 4)
- `never_load` enforcement for core components (Issue #20 — minimal mode)
- Override system mechanics (Issue #21)

## 11. Backward Compatibility

- `dotfiles_is_minimal()` stays as a thin wrapper
- `~/.dotfiles/cache/` still works (default in data dir search chain)
- No cache = all modules load (improvement: current no-cache = nothing loads)
- **Old caches need regeneration.** The new loader uses `DOTFILES_MODE_*` variables, not the old `DOTFILES_MINIMAL_TRIGGERS` / `DOTFILES_MINIMAL_MODULES`. Running `setup/generate-cache.sh` (or later, the CLI) regenerates in the new format.
