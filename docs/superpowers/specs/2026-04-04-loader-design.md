# core/loader.sh — Design Specification

**Date:** 2026-04-04
**Issue:** #3
**Depends on:** #2 (core/platform.sh)

## Purpose

The loader is the framework's toolkit — it defines functions for section guards, minimal mode detection, and module loading. It reads cached config when available and falls back to sensible defaults. The actual orchestration (what order to call things) lives in `.bash_profile` (#11).

## Bootstrap

```bash
DOTFILES_DIR    # derived from BASH_SOURCE — path to dotfiles repo
DOTFILES_MODULES_DIR="${DOTFILES_DIR}/modules"
DOTFILES_CACHE_DIR="${HOME}/.dotfiles/cache"
DOTFILES_PROFILE_CACHE="${DOTFILES_CACHE_DIR}/profile.sh"
```

`DOTFILES_DIR` is derived via `$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)`. All variables exported.

## Profile Cache (fast path)

Same pattern as platform.sh. Sources `~/.dotfiles/cache/profile.sh` if it exists. That file sets:

```bash
DOTFILES_ENABLED_MODULES=("git" "modern-tools" "containers")
DOTFILES_DISABLED_SECTIONS=("kubernetes.helm" "git.log")
DOTFILES_MINIMAL_TRIGGERS=("CLAUDE_CODE" "CODEX" "GEMINI_CLI" "OPENCODE" "GROK_CLI" "CI" "GITHUB_ACTIONS" "GITLAB_CI")
DOTFILES_MINIMAL_MODULES=("git")
```

**Fallback (no cache):** Empty enabled modules, empty disabled sections, hardcoded default minimal triggers. Phase 1 loads no modules (none exist yet) but the machinery is ready for Phase 2.

Default minimal triggers (hardcoded fallback):
`CLAUDE_CODE`, `CODEX`, `GEMINI_CLI`, `OPENCODE`, `GROK_CLI`, `CI`, `GITHUB_ACTIONS`, `GITLAB_CI`

## Functions

### dotfiles_section(section_name)

Section guard. Checks whether `section_name` is in `DOTFILES_DISABLED_SECTIONS`. Returns 0 if enabled, 1 if disabled.

```bash
dotfiles_section "git.shortcuts" && {
  alias g='git'
  alias gs='git status'
}
```

Uses array substring match: `[[ ! " ${DOTFILES_DISABLED_SECTIONS[*]} " =~ " ${section} " ]]`

### dotfiles_is_minimal()

Returns 0 if minimal mode should be active. Checks:
1. `DOTFILES_MINIMAL=true` (explicit override)
2. Any env var in `DOTFILES_MINIMAL_TRIGGERS` is non-empty

### load_module(name)

Loads a single module by name:
1. Verify `${DOTFILES_MODULES_DIR}/${name}` exists, return 1 if not
2. Check for cached module manifest (`~/.dotfiles/cache/modules/${name}.sh`) — if present, source it for load order and platform compatibility
3. Fallback: glob `*.sh` files in the module directory, source alphabetically
4. Each sourced file has section guards internally — the loader doesn't filter sections, modules do

### dotfiles_load_modules()

Iterates `DOTFILES_ENABLED_MODULES` array, calls `load_module()` for each.

### dotfiles_load_minimal_extras()

For minimal mode only. Iterates `DOTFILES_MINIMAL_MODULES` array, calls `load_module()` for each. Allows profile-specified modules (e.g., git aliases) to load even in AI/CI contexts.

## What the Loader Does NOT Do

- **Orchestration** — `.bash_profile` (#11) decides the loading sequence
- **Module discovery** — the CLI (#22+) scans modules and generates caches
- **Profile parsing** — the CLI reads `~/.dotfiles.json` and generates `profile.sh`
- **Override loading** — `.bash_profile` handles sourcing overrides after modules

## Cache Files

| File | Generated by | Contains |
|---|---|---|
| `~/.dotfiles/cache/platform.sh` | core/platform.sh fallback, CLI later | Platform variables |
| `~/.dotfiles/cache/profile.sh` | CLI (Phase 3+) | Enabled modules, disabled sections, minimal config |
| `~/.dotfiles/cache/modules/{name}.sh` | CLI (Phase 4+) | Per-module load order, platform compat |

## Testing

- `dotfiles_section` correctly gates blocks (enabled returns 0, disabled returns 1)
- `dotfiles_is_minimal` activates when trigger env vars are set
- `load_module` sources files in directory, handles missing module gracefully
- Empty state (no cache, no modules) works without errors
- All functions are pure bash, no external dependencies
